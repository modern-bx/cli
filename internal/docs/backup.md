# Bitrix backup extractor

`backup:extract` распаковывает файловую часть Bitrix backup-архива в пустую директорию назначения.

Текущая реализация намеренно использует максимально близкий к `restore.php` потоковый tar-reader вместо
старой декомпозированной реализации. Цель этого режима — совместимость с референсным распаковщиком
Bitrix, особенно для многотомных архивов.

## Использование

```bash
php bin/cli.php backup:extract <archive> <destination> [--password=...]
```

- `archive` — путь к первому тому архива или к любому пронумерованному тому; reader нормализует имя
  до первого тома по правилу `archive`, `archive.1`, `archive.2`.
- `destination` — пустая директория назначения. Если директории нет, команда создаст её.
- `--password` — пароль для Bitrix internal encrypted header.

## Что поддерживается

- обычный tar-поток;
- gzip/tgz-поток;
- multipart-тома с последовательными именами;
- GNU long-name записи;
- Bitrix internal encrypted header;
- checksum-проверка tar-заголовков;
- базовая защита от небезопасных путей и выхода за пределы destination.

## Что намеренно не переносилось из restore.php

Команда не запускает браузерный restore-flow и не выполняет действия, не относящиеся к файловой
распаковке:

- не восстанавливает базу данных;
- не перезаписывает `dbconn.php`, `.settings.php` и другие настройки подключения;
- не удаляет архивные файлы после завершения;
- не выполняет cloud/download steps;
- не модифицирует `.htaccess` специальной логикой restore.php.

## Ограничения

Это compatibility-first реализация. Она проще старого native extractor-а и не предоставляет его
расширенную политику ownership/mtime/atomic writes/incidents. Эти возможности можно возвращать позже,
после стабилизации совместимости с реальными Bitrix backup-архивами.
