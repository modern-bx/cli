# Валидация JSON-аргументов команд

В CLI для проверки сложных JSON-аргументов используется небольшой валидатор
`ModernBx\Cli\App\Validation\JsonSchemaValidator`. Схемы лежат рядом с командами
в подпапках `Validation` и остаются похожими на JSON Schema, но используют два
проектных ключа:

- `$mixin` — подключение общих файлов схем с наборами `$defs`.
- `$include` — правило для полей объекта, которых нет в `properties`.

## Типичный файл схемы

Минимальная схема аргумента команды обычно состоит из следующих секций:

```json
{
  "$schema": "https://json-schema.org/draft/2020-12/schema",
  "title": "Аргумент fields команды site:update",
  "description": "Допустимые поля JSON-объекта для обновления таблицы Bitrix b_lang.",
  "$mixin": [
    "../../Validation/Global.schema.json"
  ],
  "type": "object",
  "$include": false,
  "properties": {
    "NAME": {
      "$ref": "#/$defs/string?"
    }
  }
}
```

Назначение секций:

- `$schema` хранит версию формата для читателей и редакторов JSON Schema.
- `title` и `description` описывают назначение схемы для человека.
- `$mixin` подключает общие определения из других файлов.
- `type` задаёт базовый тип значения, которое проверяется схемой.
- `properties` описывает разрешённые поля JSON-объекта.
- `$include` описывает, что делать с полями, которых нет в `properties`.

## Подключение общих определений через `$mixin`

`$mixin` — это массив строк с путями к другим JSON-файлам. Относительные пути
считаются от директории текущей схемы.

```json
{
  "$mixin": [
    "../../Validation/Global.schema.json"
  ]
}
```

Валидатор читает каждый файл из `$mixin`, рекурсивно обрабатывает его собственные
`$mixin`-подключения и сливает секцию `$defs` в текущую схему. После этого ссылки
вида `#/$defs/name` могут использовать определения из подключённых файлов.

Валидатор дополнительно проверяет ошибки подключения:

- путь в `$mixin` должен быть непустой строкой;
- подключаемый файл должен существовать и содержать JSON-объект;
- циклические подключения запрещены;
- дублирующиеся имена `$defs` запрещены.

## Общие встроенные типы

Общие типы хранятся в `src/ModernBx/Cli/App/Console/Command/Validation/Global.schema.json`.
Сейчас доступны следующие определения:

### `string?`

Строка или `null`.

```json
{
  "$ref": "#/$defs/string?"
}
```

Используется для полей Bitrix, которые обычно передаются строками, но могут быть
обнулены.

### `numeric-string`

Строка, которую PHP считает числовой через `is_numeric()`.

```json
{
  "$ref": "#/$defs/numeric-string"
}
```

### `numeric`

Целое число или `numeric-string`. Тип намеренно не принимает boolean и `float`.

```json
{
  "$ref": "#/$defs/numeric"
}
```

### `numeric?`

`numeric` или `null`. Суффикс `?` в идентификаторах общих типов означает, что
значение можно обнулить.

```json
{
  "$ref": "#/$defs/numeric?"
}
```

### `scalar?`

Строка, целое число, `float`, boolean или `null`.

```json
{
  "$ref": "#/$defs/scalar?"
}
```

### `jsonValue`

Рекурсивное JSON-значение: `null`, boolean, integer, float, string, массив JSON-значений
или объект, у которого каждое дополнительное поле тоже является `jsonValue`.

```json
{
  "$ref": "#/$defs/jsonValue"
}
```

Этот тип нужен для произвольных JSON-структур: массивов, вложенных объектов и
рекурсивных деревьев.

## Поддерживаемые ключи схемы

Валидатор поддерживает ограниченный набор ключей, достаточный для схем аргументов
команд:

- `$mixin` — подключает внешние `$defs` из файлов.
- `$defs` — хранит именованные определения схем.
- `$ref` — ссылается на локальное определение, например `#/$defs/jsonValue`.
- `type` — проверяет тип значения.
- `enum` — ограничивает значение списком допустимых литералов.
- `anyOf` — принимает значение, если подходит хотя бы одна вложенная схема.
- `properties` — описывает известные поля объекта.
- `$include` — задаёт правило для неизвестных полей объекта.
- `items` — задаёт схему для каждого элемента массива.

## Типы значений в `type`

`type` может быть строкой:

```json
{
  "type": "string"
}
```

или массивом строк:

```json
{
  "type": [
    "string",
    "null"
  ]
}
```

Поддерживаются типы:

- `object` — JSON-объект;
- `array` — JSON-массив;
- `string` — строка;
- `integer` — целое число;
- `float` — число с плавающей точкой;
- `numeric-string` — числовая строка;
- `boolean` — boolean;
- `null` — `null`.

## Правила для объектов

Для объектных структур обычно используются `type`, `properties` и `$include`.

```json
{
  "type": "object",
  "$include": false,
  "properties": {
    "NAME": {
      "type": "string"
    },
    "SORT": {
      "type": [
        "integer",
        "string"
      ]
    }
  }
}
```

`properties` перечисляет известные поля. `$include` управляет полями, которых нет
в `properties`:

- `false` — неизвестные поля запрещены;
- объект-схема — каждое неизвестное поле валидируется по этой схеме;
- отсутствует — неизвестные поля разрешены без дополнительной проверки.

Пример объекта с произвольными JSON-значениями в дополнительных полях:

```json
{
  "type": "object",
  "$include": {
    "$ref": "#/$defs/jsonValue"
  }
}
```

## Правила для массивов

Для массива используется `type: array` и `items`.

```json
{
  "type": "array",
  "items": {
    "type": "string"
  }
}
```

Каждый элемент массива должен соответствовать схеме из `items`. В `items` можно
использовать `$ref`, `anyOf`, `type`, `enum`, вложенные `object` и любые другие
поддерживаемые ключи.

## Union-типы

Простой union можно выразить массивом типов:

```json
{
  "type": [
    "string",
    "integer",
    "null"
  ]
}
```

Если варианты отличаются не только типом, используйте `anyOf`:

```json
{
  "anyOf": [
    {
      "type": "string"
    },
    {
      "type": "object",
      "$include": false,
      "properties": {
        "id": {
          "type": "integer"
        }
      }
    }
  ]
}
```

Значение считается валидным, если оно подошло хотя бы под одну схему из `anyOf`.

## Intersection-типы

Отдельного ключа `allOf` сейчас нет. Intersection в текущем валидаторе собирается
композицией ограничений внутри одной схемы: `type`, `enum`, `properties`,
`$include`, `items` и `$ref` применяются совместно.

Например, объект должен одновременно быть объектом, иметь разрешённую структуру
из `properties` и запрещать неизвестные поля:

```json
{
  "type": "object",
  "$include": false,
  "properties": {
    "ACTIVE": {
      "enum": [
        "Y",
        "N"
      ]
    }
  }
}
```

Если понадобится полноценная композиция нескольких независимых схем как в
`allOf`, её нужно добавить в валидатор отдельно.

## Вложенные структуры

Схемы можно вкладывать друг в друга напрямую:

```json
{
  "type": "object",
  "$include": false,
  "properties": {
    "filter": {
      "type": "object",
      "$include": {
        "$ref": "#/$defs/jsonValue"
      }
    },
    "select": {
      "type": "array",
      "items": {
        "type": "string"
      }
    }
  }
}
```

Так описываются структуры вида `object -> array -> object` любой глубины.

## Рекурсия

Рекурсия задаётся через `$ref` на определение из `$defs`. Пример — общий тип
`jsonValue`:

```json
{
  "$defs": {
    "jsonValue": {
      "anyOf": [
        {
          "type": "string"
        },
        {
          "type": "array",
          "items": {
            "$ref": "#/$defs/jsonValue"
          }
        },
        {
          "type": "object",
          "$include": {
            "$ref": "#/$defs/jsonValue"
          }
        }
      ]
    }
  }
}
```

Массив ссылается на `jsonValue` для каждого элемента, а объект ссылается на
`jsonValue` для каждого дополнительного поля. Так формируется рекурсивное дерево
JSON-значений.

## Ошибки валидации

`JsonSchemaValidator::validate()` возвращает список строк с ошибками. Если список
пустой, значение валидно. Команда сама решает, как показать эти ошибки
пользователю. Например, `site:update` объединяет ошибки через `PHP_EOL` и
выбрасывает исключение с локализованным сообщением.

Ошибки схемы или `$mixin`-подключений выбрасываются как `InvalidArgumentException`.
Команды должны перехватывать их и превращать в пользовательское сообщение о
некорректной схеме.

## Рекомендации по добавлению новых схем

1. Кладите схему в подпапку `Validation` рядом с командой.
2. Называйте файл по команде и аргументу, например `SiteUpdateFields.schema.json`.
3. Подключайте общие типы через `$mixin`, если нужны `$defs` из глобальной схемы.
4. Для закрытых объектов указывайте `"$include": false`.
5. Для произвольных JSON-объектов используйте `"$include": { "$ref": "#/$defs/jsonValue" }`.
6. Для union-типов сначала пробуйте массив `type`, а для сложных вариантов — `anyOf`.
7. Для intersection-типа объединяйте ограничения в одной схеме; если этого мало,
   расширяйте валидатор новым ключом.
